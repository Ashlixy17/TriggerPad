# CS2 GSI 实现路径（草案）

## 1. 文档目的

本文档用于说明 CS2 GSI 的初步实现顺序，作为新项目启动时的方向参考。

当前版本只确定主要阶段，不限定编程语言、框架、目录结构和完整事件范围。实现过程中可根据真实 GSI 数据、测试结果和 TriggerPad 的需求继续调整。

当前开发以 TriggerPad 的实际需求为优先，不为了潜在的通用场景提前扩大首版范围。

## 2. 范围

- 仅使用 Valve 提供的 Game State Integration（GSI）。
- 接收和解析 CS2 主动发送的 GSI 数据。
- 将状态变化转换为 TriggerPad 可使用的统一事件。
- 不读取游戏内存，不注入或修改游戏进程。
- 不开发通用 CS2 SDK。

## 3. 实现原则

- 先完成独立的 GSI 核心能力，再接入 TriggerPad。
- GSI 核心负责配置、Payload 解析、状态快照、状态差分和统一事件。
- GSI 核心不依赖 TriggerPad 的 UI、规则引擎和音频模块。
- TriggerPad 通过适配层消费 GSI 核心输出的事件。
- 服务层暂不开发；未来有明确需求时，再基于稳定的核心层提供 HTTP、WebSocket 或其他接口。
- 当前只保持清晰边界，不提前设计插件系统、跨语言协议或公共 SDK API。

## 4. 实现路径

### 阶段一：验证 GSI 通路

- 创建最小 GSI 配置文件。
- 启动本地 HTTP 接收端。
- 确认 CS2 能够发送请求并保存原始 JSON。
- 记录不同游戏状态下的真实数据样本。

**阶段结果：** 可以稳定收到并查看 CS2 的原始 GSI 数据。

### 阶段二：整理 Payload

- 根据真实样本整理主要数据区域和字段。
- 区分必有字段、可选字段和不同场景下才会出现的字段。
- 对未知或缺失字段采用宽松处理，避免解析失败导致程序中断。

**阶段结果：** 形成可继续调整的 GSI 数据模型和样本集合。

### 阶段三：建立状态快照

- 将每次收到的 Payload 转换为当前状态快照。
- 保留上一份有效状态，用于比较前后变化。
- 明确应用启动、进入比赛、退出比赛和长时间未收到数据时的状态。

**阶段结果：** 系统能够维护连续、可判断有效性的游戏状态。

### 阶段四：转换为统一事件

- 比较前后状态，识别有意义的变化。
- 优先实现 TriggerPad MVP 实际需要的回合、玩家、武器和炸弹事件。
- 对重复 Payload、缺失状态和场景切换进行去重或保护。
- 让上层只依赖统一事件，不直接依赖原始 GSI JSON。

**阶段结果：** GSI 状态变化可以稳定转换为 TriggerPad 可消费的事件。

### 阶段五：补充可靠性

- 验证请求来源或配置中的认证信息。
- 限制请求大小并处理错误 JSON。
- 增加必要的运行状态、错误和诊断日志。
- 确保接收和事件处理不会阻塞 UI 或其他模块。

**阶段结果：** 异常输入、游戏退出或配置错误不会导致程序崩溃。

### 阶段六：建立测试能力

- 将脱敏后的真实 Payload 保存为测试样本。
- 支持使用样本模拟和回放 GSI 数据。
- 测试常见状态转换、重复消息、字段缺失和异常输入。

**阶段结果：** 无需每次启动 CS2，也能验证主要 GSI 行为。

### 阶段七：接入 TriggerPad

- 确定 GSI 模块向 Event System 输出的最小接口。
- 将统一事件接入规则匹配和日志链路。
- 验证从收到 GSI 到触发动作的完整流程和延迟。

**阶段结果：** TriggerPad 能够根据受支持的 CS2 GSI 事件稳定触发规则。

## 5. 初期完成标准

- 能自动或手动完成 GSI 配置。
- 能稳定接收并解析真实 Payload。
- 能维护前后状态并产生一组 MVP 事件。
- 异常或不完整数据不会使程序崩溃。
- 能使用保存的样本完成基本回放测试。
- 能与 TriggerPad Event System 完成最小闭环。
