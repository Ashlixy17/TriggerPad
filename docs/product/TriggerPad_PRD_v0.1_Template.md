# 产品理念

> **Every Trigger Drives an Action.**

TriggerPad 是一款基于 **Trigger → Rule → Action** 模型构建的事件驱动自动化工具。

第一阶段聚焦 Counter-Strike 2（以下简称“CS2”）：通过 Valve Game State Integration（GSI）监听游戏状态和事件，并根据用户配置自动执行指定动作；MVP 的主要动作为播放音频。

产品未来可扩展更多触发器、动作类型及游戏平台，但所有能力均应遵守产品边界，不得影响游戏公平性或违反相关平台规则。

# 文档目录

## Part A 项目定义

### 01 项目背景

Soundpad 等音频工具通常需要用户手动按下提前设置好的快捷键，才能通过麦克风播放音频。在游戏过程中，手动操作会打断游戏节奏，也难以与实时游戏事件准确同步，难以达到微妙的效果。

TriggerPad 希望通过读取游戏实时状态，让用户预先配置“事件—规则—音频”，在满足条件时自动播放指定音频，从而提供更自然的多人娱乐体验。

后续版本可在安全边界内扩展自定义动作，例如在角色死亡后打开指定网页。此类非音频动作不属于 MVP 范围。

### 02 产品定位

TriggerPad 是一款面向 CS2 玩家的娱乐型自动化音频工具，核心价值是将游戏事件与音频播放自动关联。后续可能将其功能拓展至其他支持的游戏。

产品仅提供娱乐及效率辅助能力，不提供透视、瞄准、对局分析提示等任何可能影响竞技公平性的功能和自动化脚本。

### 03 产品理念

- **娱乐优先：** 提升组队和多人对局中的互动体验。
- **公平合规：** 不读取或利用可形成竞技优势的信息，不模拟作弊行为。
- **配置直观：** 用户无需编写代码即可创建自动化规则。
- **状态透明：** 触发、执行、失败和恢复过程均可查看。
- **安全可控：** 用户可随时停用规则或关闭自动执行。

### 04 用户画像

**核心用户：**

- 喜欢在 CS2 多人对局或五排场景中进行语音互动、整活的玩家。
- 希望减少快捷键操作，让音频与游戏事件自动同步的玩家。

**扩展用户：**

- 有自定义事件触发需求、愿意配置规则的进阶玩家。
- 未来希望在其他游戏中使用类似自动化能力的玩家。

## Part B 产品规划

### 05 MVP 范围

MVP 应形成一条完整且可验证的使用链路：

1. 完成首次启动检测，并确认 CS2、GSI 与音频输出环境可用。
2. 从 GSI 接收并解析受支持的游戏事件。
3. 在事件树中选择触发事件。
4. 将本地音频导入音频池。
5. 创建“事件 → 规则 → 播放音频”的自动化规则。
6. 在游戏事件发生时，通过选定的音频设备或虚拟麦克风播放音频。
7. 在日志中查看事件接收、规则匹配和动作执行结果。
8. 提供规则启用、停用、编辑和删除能力。

MVP 暂不包含：其他游戏适配、云端同步、在线音频市场、脚本系统，以及浏览器或第三方应用等非音频动作。

### 06 产品边界

- 仅处理 Valve GSI 正常提供且产品明确支持的数据。
- 不读取游戏内存，不注入游戏进程，不修改游戏文件或网络数据。
- 不提供可形成竞技优势的提示、预测或自动操作能力。
- 音频内容由用户自行导入和管理；用户应对内容版权及使用方式负责。
- 软件应提供总开关、规则开关及异常停止机制，避免失控触发或连续播放。
- 产品不承诺所有第三方虚拟音频设备均可兼容；支持范围以测试结果为准。

### 07 软件逻辑架构

MVP 的逻辑链路如下：

`CS2 GSI → Event System → Rule Engine → Audio System → Audio Router → 输出设备/虚拟麦克风`

- **Event System：** 接收、解析并标准化 GSI 数据。
- **Rule Engine：** 判断事件是否满足用户配置的触发条件。
- **Audio System：** 管理音频资源、播放状态和播放队列。
- **Audio Router：** 将音频路由至用户选择的输出设备。
- **Log & Health Check：** 记录链路状态，并提供问题定位和恢复建议。

具体技术选型与模块接口在开发设计文档或 ADR 中确定。

## Part C UI 设计

### 08 UI 设计原则

- 优先展示“事件、规则、音频”三者之间的关系，降低理解成本。
- 关键状态明确可见，包括连接状态、规则启停、音频设备和执行结果。
- 创建规则时提供即时校验，避免保存缺少事件或音频的无效规则。
- 错误信息应说明问题、可能原因及建议操作，不仅展示错误代码。
- 危险或不可逆操作（如删除规则、卸载虚拟麦克风）需二次确认。
- 深色模式与 CS2 使用场景保持一致，同时保证文字和状态颜色可辨识。

### 09 初始化向导

首次启动时，引导用户完成运行环境检测和基础配置：

1. 检测 Steam 与 CS2 的安装或运行状态。
2. 检查或引导配置 CS2 GSI。
3. 检测可用的音频输入、输出及虚拟麦克风设备。
4. 播放测试音频，验证音频链路是否正常。
5. 展示检测结果，并对异常项提供重试、跳过或修复指引。

用户跳过非阻塞项后仍可进入主界面，并可在 Health Check 中重新检测。

### 10 主界面

主界面顶部标签栏提供以下页面切换：

- **快速开始：** 展示事件树、规则区和音频池，是主要配置与操作界面。
- **日志：** 展示 GSI 事件、规则匹配、动作执行及错误信息，并支持基础筛选。
- **设置：** 提供主题、更新、音频设备及虚拟麦克风相关设置。

快速开始页应突出当前连接状态和全局启停开关。用户完成事件选择、音频选择和规则配置后，可在同一页面保存并启用规则。

## Part D 核心功能

### 11 Event Tree

事件树展示可从 GSI 读取且可用于自动触发的游戏事件。

- 事件按类别以树状结构排列，支持展开、折叠和搜索。
- 用户选择事件后，可将其添加至规则区作为触发条件。
- 不可用或当前版本不支持的事件应隐藏，或明确标注为不可用。
- 每个事件应提供简短说明，必要时展示可配置的条件参数。

### 12 Rule Editor（规则区）

规则区用于建立和管理“事件 → 条件 → 动作”的对应关系。

- 用户可选择触发事件、配置支持的条件并指定音频动作。
- 规则支持命名、启用、停用、编辑和删除。
- 保存前校验必要字段，并提示重复规则或潜在冲突。
- 应支持基础防抖或冷却时间，避免同一事件短时间内重复触发音频。
- 执行失败时记录日志，不得导致主程序崩溃或阻塞其他有效规则。

### 13 Sound Pool

音频池用于管理可被规则引用的本地音频文件。

- 支持通过文件选择器或拖放方式导入音频。
- 展示音频名称、时长、格式及可用状态。
- 支持试听、重命名显示名称和移除音频。
- 删除已被规则引用的音频前，应提示受影响的规则。
- 支持的文件格式、大小及时长限制由技术验证后确定，并在界面中明确说明。

### 14 Trigger Library

MVP 阶段由内置的 CS2 GSI 事件构成触发器库，不提供用户脚本或第三方触发器扩展能力。

后续版本可评估触发器模板、第三方游戏适配和扩展机制，具体方案待定。

### 15 Action Library

MVP 阶段仅提供“播放音频”动作。

打开网页、启动应用等扩展动作属于后续规划项；引入前需评估权限、安全风险及误触发处理机制。

### 16 核心功能通用验收维度

以上核心功能统一按以下维度设计与评审：

- 设计目标（Why）
- 用户价值（Value）

- 状态设计（State）

- 非目标（Out of Scope）
- 验收标准（Acceptance Criteria）

## Part E 系统设计

### 17 Rule Engine

Rule Engine 接收标准化事件，按已启用规则进行匹配，并将符合条件的动作提交给执行模块。

MVP 至少应处理规则启停、条件校验、冷却时间、冲突提示和执行结果记录。规则优先级及并发策略由技术设计进一步明确。

### 18 Event 系统

Event 系统负责接收 GSI 数据、校验请求、解析事件并转换为内部统一格式。异常或不完整数据应被安全忽略并记录诊断信息，不得中断主程序。

### 19 Audio 系统

Audio 系统提供与 Soundpad 类似的本地音频播放能力，并支持通过虚拟麦克风将音频发送至游戏语音链路。

系统应管理音频加载、试听、播放、停止、队列和失败状态。多条规则同时触发时的混音或排队策略待技术验证后确定。

### 20 Audio Router

Audio Router 负责枚举音频设备、保存用户选择并将音频输出到指定设备。设备不可用、被占用或发生变更时，应停止相关动作、记录错误并提示用户重新选择设备。

## Part F 工具

### 21 Developer Tools

面向开发和调试场景，提供模拟 GSI 事件、查看原始事件数据及测试规则执行链路的能力。该入口默认不面向普通用户展示。可以不作为MVP的验收标准

### 22 Health Check

Health Check 用于检测当前电脑及应用运行状态，包括：

- Steam、CS2 与 GSI 配置状态。
- 音频输出设备及虚拟麦克风可用状态。
- 测试音频播放结果。
- 规则缺失、引用失效及潜在冲突。
- 当前版本及更新状态。

检测结果应区分“正常、警告、错误”，并提供明确的处理建议。

### 23 Latency Benchmark

用于测量从接收到 GSI 事件到开始播放音频的端到端延迟，并输出可供排查的分段耗时。该能力是否进入 MVP，取决于开发与测试成本评估。

### 24 Replay

用于回放已记录或模拟的 GSI 事件，以验证规则匹配和动作执行。不作为MVP的验收标准，后期可开发作为开发测试工具，是否面向普通用户开放待定。

### 25 Auto Fix

当音频播放未达到预期时，系统可尝试执行低风险修复，例如重新初始化音频设备或刷新设备列表。自动修复失败时，应停止重复尝试、生成错误日志，并引导用户打开 Health Check。

涉及安装、卸载或系统权限的操作不得静默执行，必须由用户确认。

## Part G 设置

### 26 Settings

基础设置包括：

- 深色/浅色主题。
- 开机启动与最小化行为。
- 音频输入、输出及虚拟麦克风设备选择。
- 全局音量、规则总开关及通知偏好。
- 日志级别及数据保留设置。

### 27 Data Manager

用于管理规则配置、音频索引和日志数据，支持查看存储位置、清理日志，以及导入/导出配置。涉及本地音频文件时，应明确说明导出包是否包含原始音频。

### 28 Update

用户可选择自动检查更新，也可手动检查更新。更新前应展示版本号和更新说明；更新失败不得影响当前版本继续使用，并应提供重试或手动下载指引。

## Part H 项目规划

### 29 Performance

MVP 重点关注以下指标：

- 软件空闲状态下不应持续占用大量 CPU 或内存。
- GSI 事件处理和规则匹配不应阻塞 UI。
- 常规场景下，事件到音频开始播放的延迟应稳定且用户可感知地接近实时。
- 长时间运行后不得出现明显的内存持续增长、事件堆积或重复播放。

具体量化阈值应在原型测试后补充，并作为发布验收依据。

### 30 Roadmap

**阶段 1：MVP**

- 完成 CS2 GSI 接入。
- 完成事件树、规则编辑器、音频池和音频输出链路。
- 完成初始化向导、日志和基础 Health Check。

**阶段 2：体验完善**

- 优化诊断、自动修复、配置导入导出和延迟测试。
- 根据用户反馈扩充可用事件和规则条件。

**阶段 3：能力扩展**

- 评估更多游戏、触发器及动作类型。
- 评估安全可控的扩展机制。

路线图不代表交付承诺，具体排期由项目资源和验证结果决定。

### 31 开发规范

- 开发任务应关联明确的 PRD 条目或已确认的需求变更。
- 新功能应包含必要的测试、错误处理和日志。
- 不得引入超出产品边界或影响游戏公平性的实现。
- 关键技术决策应通过 ADR 记录背景、方案和取舍。
- PRD、设计、实现和测试验收口径应保持一致。

### 32 GitHub 规划

- 使用 Issue 跟踪需求、缺陷和技术任务。
- 使用 Pull Request 进行代码审查，并关联对应 Issue 或 PRD 条目。
- 使用 Milestone 管理 MVP 及后续版本范围。
- 使用标签区分功能、缺陷、文档、测试和优先级。
- 仓库公开范围、贡献指南及许可证方案待项目负责人确认。

### 33 MVP 验收标准

MVP 发布前至少满足以下条件：

1. 用户可完成首次启动检测并查看每项检测结果。
2. 软件可稳定接收并解析受支持的 CS2 GSI 事件。
3. 用户可导入、试听和移除受支持的本地音频。
4. 用户可创建、编辑、启停和删除“事件 → 播放音频”规则。
5. 受支持事件发生且规则满足条件时，指定音频可通过选定设备播放。
6. 同一事件的重复触发可通过冷却时间得到控制。
7. 事件接收、规则匹配、动作成功或失败均有可查看的日志。
8. 音频设备不可用、规则引用失效或 GSI 断开时，软件能给出明确提示且不崩溃。
9. 用户可通过全局开关立即停止自动执行。
10. 产品不读取游戏内存、不注入游戏进程，也不提供影响竞技公平性的功能。

具体兼容系统、音频格式、性能阈值及稳定性测试时长，应在发布测试计划中补充。

## Part I 附录

### 34 Glossary

- **Trigger：** 触发器，表示可被系统识别并用于启动规则判断的事件。
- **Rule：** 规则，描述触发器、条件和动作之间的关系。
- **Action：** 动作，规则满足后由系统执行的操作；MVP 中为播放音频。
- **GSI：** Game State Integration，Valve 提供的游戏状态数据接口。
- **SSOT：** Single Source of Truth，统一且权威的信息来源。
- **虚拟麦克风：** 将本地音频作为麦克风输入提供给其他应用的虚拟音频设备。

### 35 Concept Model

核心概念关系如下：

`Event/Trigger → Rule（Condition + Cooldown）→ Action（Play Audio）→ Execution Log`

- 一个触发器可被多条规则引用。
- 一条规则在 MVP 中关联一个主要触发器和一个音频动作。
- 一个音频资源可被多条规则引用。
- 每次规则判断及动作执行均应生成可追踪的日志记录。

具体字段、标识符和持久化方案由技术设计确定。

### 36 ADR（Architecture Decision Record）

对影响系统结构、兼容性、安全性或长期维护成本的重要技术决策，应建立 ADR。每份 ADR 至少包含背景、候选方案、最终决策、决策理由及影响范围。
