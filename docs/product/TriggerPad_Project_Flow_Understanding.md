# TriggerPad 项目流程理解

> 本文档基于《TriggerPad PRD v0.1（框架草案）》整理，用于说明当前项目的产品目标、用户流程、运行机制、模块职责、异常处理及 MVP 边界。
>
> 本文档是对 PRD 的辅助理解，不替代 PRD；如两者存在冲突，应以最新 PRD 为准。

## 1. 项目概述

TriggerPad 是一款面向 Counter-Strike 2（以下简称“CS2”）玩家的事件驱动自动化音频工具。

它通过 Valve Game State Integration（GSI）接收游戏状态和事件，根据用户预先配置的规则判断是否触发，并将指定的本地音频路由到音频输出设备或虚拟麦克风。同时，系统会记录事件接收、规则匹配和动作执行结果，方便用户了解当前状态并排查问题。

TriggerPad 的核心模型是：

`Trigger（触发器）→ Rule（规则）→ Action（动作）`

在 MVP 阶段，该模型具体表现为：

`CS2 GSI 事件 → 条件与冷却时间判断 → 播放本地音频`

TriggerPad 不只是一个音频播放器，而是一个以“自动播放音频”为首个落地场景的事件驱动自动化工具。

## 2. 产品目标

TriggerPad 希望解决 Soundpad 等传统音频工具必须依赖用户手动按下快捷键的问题。

在游戏过程中，手动触发音频可能打断操作，也难以与实时游戏事件准确同步。TriggerPad 允许用户预先建立游戏事件与音频之间的关系，使音频能够在指定事件发生时自动播放，从而形成更自然、更及时的多人娱乐互动。

产品遵循以下原则：

- **娱乐优先：** 服务于组队、五排和多人对局中的语音互动及娱乐体验。
- **公平合规：** 不提供任何可能形成竞技优势的功能。
- **配置直观：** 用户无需编写代码即可建立自动化规则。
- **状态透明：** 事件、规则、动作、失败和恢复过程均可查看。
- **安全可控：** 用户可以随时停用规则或停止全部自动执行。

## 3. 核心用户流程

### 3.1 首次启动与环境检测

用户第一次启动 TriggerPad 时，系统通过初始化向导完成运行环境检查和基础配置。

主要流程如下：

1. 检测 Steam 与 CS2 的安装或运行状态。
2. 检查 CS2 GSI 是否已正确配置，必要时引导用户完成配置。
3. 枚举系统中的音频输入、音频输出和虚拟麦克风设备。
4. 让用户选择目标音频设备。
5. 播放测试音频，验证从 TriggerPad 到目标设备的音频链路。
6. 展示检测结果，并区分正常、警告和错误状态。
7. 对异常项提供重试、跳过或修复指引。

非阻塞问题可以暂时跳过。用户进入主界面后，可通过 Health Check 再次执行检测。

### 3.2 导入和管理音频

用户进入“快速开始”页面后，可以通过文件选择器或拖放方式将本地音频导入 Sound Pool。

音频池负责：

- 读取并展示音频名称、格式和时长。
- 判断音频文件是否可用。
- 提供音频试听。
- 允许用户修改音频的显示名称。
- 维护音频与规则之间的引用关系。
- 在删除音频前提示受影响的规则。

MVP 使用本地音频资源，不包含云端音频同步或在线音频市场。

### 3.3 选择触发事件

Event Tree 展示 TriggerPad 当前支持的 CS2 GSI 事件。

用户可以：

1. 按类别展开或折叠事件树。
2. 搜索目标事件。
3. 查看事件说明及可配置参数。
4. 选择事件并将其添加到规则区。

事件树只展示或启用当前版本明确支持的事件。不可用事件应隐藏，或在界面中明确标注为不可用。

### 3.4 创建自动化规则

Rule Editor 用于建立“事件、条件和动作”之间的关系。

创建规则的基本流程如下：

1. 选择一个主要触发事件。
2. 配置该事件支持的判断条件。
3. 从音频池中选择需要播放的音频。
4. 设置冷却时间或防抖参数，避免短时间内重复播放。
5. 为规则命名。
6. 保存并启用规则。

规则支持启用、停用、编辑和删除。保存前，系统应检查：

- 是否缺少触发事件。
- 是否缺少音频动作。
- 引用的音频是否有效。
- 是否存在重复规则。
- 是否存在潜在规则冲突。

### 3.5 运行时自动触发

规则启用后，TriggerPad 进入持续监听和执行状态。

运行时主链路为：

`CS2 GSI → Event System → Rule Engine → Audio System → Audio Router → 输出设备/虚拟麦克风`

具体过程如下：

1. CS2 通过 GSI 向 TriggerPad 提供游戏状态数据。
2. Event System 接收并校验原始数据。
3. Event System 将原始数据转换成内部统一事件。
4. Rule Engine 查找引用该事件的已启用规则。
5. Rule Engine 判断规则条件和冷却时间是否满足。
6. 匹配成功后，Rule Engine 向 Audio System 提交播放动作。
7. Audio System 加载音频，并根据播放策略处理播放或排队。
8. Audio Router 将音频发送到用户选择的输出设备或虚拟麦克风。
9. 游戏通过虚拟麦克风接收到音频，使其进入游戏语音链路。
10. 系统记录事件接收、规则判断、动作执行及最终结果。

未满足条件的规则不会执行动作，但可以根据日志策略保留规则未匹配信息。

## 4. 系统模块职责

### 4.1 Event System

Event System 是游戏数据进入 TriggerPad 的入口，主要负责：

- 接收 CS2 GSI 数据。
- 校验数据来源和内容完整性。
- 解析游戏状态及事件。
- 将原始数据标准化为内部统一事件。
- 忽略异常或无法识别的数据。
- 记录必要的诊断信息。

Event System 不负责决定是否播放音频，只负责提供可靠、统一的事件输入。

### 4.2 Rule Engine

Rule Engine 是 TriggerPad 的核心决策模块，主要负责：

- 加载并管理用户规则。
- 过滤未启用或无效规则。
- 匹配触发事件。
- 判断规则条件。
- 处理冷却时间和防抖。
- 检查重复规则或潜在冲突。
- 将满足条件的动作提交给执行模块。
- 记录规则判断和执行结果。

规则优先级、多条规则并发触发以及冲突处理策略仍需在技术设计中进一步确定。

### 4.3 Audio System

Audio System 负责本地音频资源和播放过程，主要包括：

- 加载音频文件。
- 提供音频试听。
- 开始或停止播放。
- 管理播放状态。
- 处理播放队列。
- 捕获并上报播放失败。

多条规则同时触发时，应采用排队、打断还是混音，目前仍需通过产品和技术验证确定。

### 4.4 Audio Router

Audio Router 负责将 Audio System 产生的音频发送到指定设备，主要包括：

- 枚举可用音频设备。
- 保存用户选择的设备。
- 将音频发送到普通输出设备或虚拟麦克风。
- 检测设备被移除、被占用或不可用的情况。
- 在设备异常时停止相关动作并提示用户重新选择。

### 4.5 Log

日志系统贯穿整个运行链路，应至少记录：

- GSI 数据是否成功接收。
- 原始数据是否成功解析。
- 产生了哪个标准化事件。
- 哪些规则参与匹配。
- 规则是否满足条件。
- 动作是否成功提交和执行。
- 音频最终是否成功播放。
- 异常发生在哪个模块及可能原因。

日志既服务普通用户排查问题，也服务开发者调试系统。

### 4.6 Health Check

Health Check 用于集中检测环境、配置和运行状态，包括：

- Steam、CS2 及 GSI 状态。
- 音频输出设备和虚拟麦克风状态。
- 测试音频播放结果。
- 规则缺失、引用失效及潜在冲突。
- 当前版本和更新状态。

检测结果应区分正常、警告和错误，并提供用户可以执行的处理建议。

### 4.7 Auto Fix

Auto Fix 是受限制的低风险恢复机制，不是无限重试机制。

它可以尝试：

- 刷新音频设备列表。
- 重新初始化音频设备。
- 重新执行不涉及权限变更的安全检测。

如果自动修复失败，系统应停止重复尝试、生成错误日志，并引导用户进入 Health Check。

安装、卸载虚拟设备或其他需要系统权限的操作，不得静默执行，必须得到用户确认。

## 5. UI 信息架构

TriggerPad 主界面分为三个主要页面。

### 5.1 快速开始

快速开始是核心操作页面，集中展示：

- 当前 GSI 和音频设备连接状态。
- 全局启停开关。
- Event Tree。
- Rule Editor。
- Sound Pool。

用户应尽可能在同一页面完成事件选择、音频选择、规则创建、保存和启用。

### 5.2 日志

日志页面展示：

- GSI 事件。
- 规则匹配过程。
- 动作执行结果。
- 错误和恢复信息。

页面应提供基础筛选，帮助用户快速定位某个事件、规则或错误。

### 5.3 设置

设置页面负责管理：

- 深色或浅色主题。
- 开机启动和最小化行为。
- 音频输入、输出和虚拟麦克风设备。
- 全局音量和规则总开关。
- 通知偏好。
- 日志级别和数据保留时间。
- 自动或手动检查更新。

## 6. 系统状态划分

TriggerPad 需要同时维护三类状态。

### 6.1 环境状态

- Steam 是否可用。
- CS2 是否可用。
- GSI 是否正确配置和连接。
- 音频设备是否存在且可使用。
- 虚拟麦克风是否可用。

### 6.2 配置状态

- 已导入的音频资源和音频索引。
- 用户创建的规则。
- 规则条件与冷却时间。
- 规则和全局开关状态。
- 用户选择的音频设备。
- 应用设置和通知偏好。

### 6.3 运行状态

- 最近接收的 GSI 事件。
- 当前参与匹配的规则。
- 当前播放或排队的音频。
- 最近的动作执行结果。
- 当前错误、警告和恢复状态。

## 7. 异常处理流程

TriggerPad 的异常处理原则是“局部失败、明确提示、保留诊断信息，不中断主程序”。

### 7.1 GSI 数据异常

- 忽略不完整或无法识别的数据。
- 记录诊断信息。
- 不因单次异常中断后续事件接收。

### 7.2 规则配置异常

- 保存规则前进行字段和引用校验。
- 音频引用失效时将规则标记为异常或不可执行。
- 对重复规则和潜在冲突给出提示。

### 7.3 音频设备异常

- 设备失效、被占用或发生变更时停止相关动作。
- 记录错误原因。
- 提示用户重新选择设备。
- 必要时尝试低风险自动修复。

### 7.4 音频播放异常

- 捕获播放失败状态。
- 不阻塞其他有效规则。
- 尝试重新初始化设备或刷新设备列表。
- 修复失败后停止重复尝试并生成错误日志。

### 7.5 紧急停止

用户可以通过全局开关立即停止自动执行，避免规则失控触发或连续播放。

## 8. 数据与配置管理

Data Manager 负责管理：

- 规则配置。
- 音频资源索引。
- 应用设置。
- 执行和错误日志。
- 配置导入与导出。
- 日志清理及数据保留。

音频文件由用户在本地管理。导出配置时，需要明确导出包是否包含原始音频，避免用户误以为配置文件已经包含全部音频资源。

MVP 不包含云端同步，所有核心配置和资源关系应以本地持久化为主。

## 9. MVP 范围

MVP 必须打通以下闭环：

`首次检测 → 导入音频 → 选择事件 → 创建规则 → 接收 GSI 事件 → 匹配规则 → 播放音频 → 查看日志`

MVP 至少包括：

- 初始化向导和基础环境检测。
- CS2 GSI 接入和事件解析。
- Event Tree。
- Rule Editor。
- Sound Pool。
- 本地音频播放。
- 音频设备或虚拟麦克风路由。
- 日志。
- 基础 Health Check。
- 规则启用、停用、编辑和删除。
- 冷却时间或基础防抖。
- 全局停止开关。

## 10. MVP 非目标

MVP 暂不包含：

- 其他游戏适配。
- 云端同步。
- 在线音频市场。
- 用户脚本系统。
- 第三方触发器扩展。
- 打开网页、启动应用等非音频动作。
- 面向普通用户的 Replay 工具。
- 完整 Developer Tools。

Latency Benchmark 是否进入 MVP，取决于开发和测试成本评估。

## 11. 安全与合规边界

TriggerPad 只处理 Valve GSI 正常提供且产品明确支持的数据。

产品不得：

- 读取游戏内存。
- 注入游戏进程。
- 修改游戏文件或网络数据。
- 提供透视、瞄准、预测或对局分析提示。
- 自动执行可能形成竞技优势的游戏操作。
- 静默执行需要系统权限的安装或卸载操作。

音频内容由用户自行导入和管理，用户需要对音频版权和使用方式负责。

## 12. 项目路线图

### 阶段 1：MVP

- 完成 CS2 GSI 接入。
- 完成事件树、规则编辑器和音频池。
- 完成音频输出及虚拟麦克风链路。
- 完成初始化向导、日志和基础 Health Check。

### 阶段 2：体验完善

- 完善问题诊断与低风险自动修复。
- 增加配置导入和导出。
- 完善延迟测试。
- 根据用户反馈增加事件和规则条件。

### 阶段 3：能力扩展

- 评估其他游戏适配。
- 评估更多触发器和动作类型。
- 评估安全可控的扩展机制。

## 13. 当前待明确事项

PRD 已经明确产品目标和主要流程，但进入详细技术设计和开发前，仍需确定以下事项：

1. MVP 首批支持的具体 GSI 事件清单。
2. 每类事件支持的条件参数。
3. 支持的音频格式、大小和时长限制。
4. 多条规则同时触发时采用排队、打断还是混音。
5. 规则优先级和冲突判定方式。
6. 虚拟麦克风由软件提供、引导安装，还是依赖第三方设备。
7. 支持的 Windows 版本及音频设备兼容范围。
8. 事件到音频开始播放的目标延迟。
9. 规则、音频索引、设置和日志的持久化格式。
10. GSI 断开后的重连和状态恢复策略。
11. 软件升级方式、失败回滚和配置兼容策略。
12. 日志默认保留时间及隐私处理方式。

这些事项应通过后续技术设计文档、测试计划或 ADR 逐项确认。

## 14. 总结

TriggerPad 的 MVP 是一条围绕 CS2 GSI 和本地音频构建的自动化执行链路：用户配置事件、规则和音频，系统在收到游戏事件后完成匹配并自动播放，同时提供日志、健康检查、异常恢复和全局停止能力。

项目当前最重要的目标不是扩展更多动作，而是先保证以下核心链路稳定、低延迟、可诊断且不会失控：

`事件接收 → 规则匹配 → 音频执行 → 结果记录`

在该闭环稳定后，再逐步扩展更多事件、更多游戏和更多动作类型。
